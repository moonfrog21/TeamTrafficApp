# Product Requirements Document (PRD): GPS Telemetry Application

## 1. Project Overview and Objectives
This document outlines the software requirements for a streamlined, cross-platform mobile application developed in Flutter for iOS and Android environments. The primary functional objective is to capture user location telemetry and transmit it securely to a dynamic server endpoint tunneled through ngrok. The application prioritizes absolute operational simplicity, restricting the main interface to a single 
data-transmission action and a secondary configuration menu. By centralizing the network configuration into a persistent settings menu, the system guarantees resilience against the frequent domain rotations inherent to free ngrok accounts.

## 2. Functional Requirements & User Interface
The primary user interface must feature a highly visible button labeled strictly as "Send GPS data" that initiates the data extraction and networking sequence. Prior to extraction, the application must explicitly request and secure foreground location permissions utilizing native iOS and Android privacy paradigms. A secondary UI element, represented by a standard settings cog icon, must navigate the user to a configuration screen containing a text input field. This text field allows the user to manually define the base ngrok domain (e.g., "https://reps-flavoring-subarctic.ngrok-free.dev"), which the application will actively validate for basic URL formatting correctness.

## 3. Networking and Payload Construction
Upon execution of the transmission button, the software must generate an HTTP POST request targeting the concatenation of the user-defined base URL and the `/data` endpoint. The networking payload must precisely mirror the defined schema, serializing as a JSON object containing the keys `user`, `latitude`, `longitude`, and `time`. The `latitude` and `longitude` values must be dynamically populated from the active GPS sensor, while the `time` value must reflect the device's localized system clock at the exact moment of transmission. The system must natively catch HTTP exceptions, specifically `ConnectionError` equivalents, ensuring the application does not crash if the target ngrok tunnel is offline or unresponsive.

## 4. Data Persistence and State Management
To guarantee seamless operation across application restarts and device power cycles, the user-defined base URL must be committed to the device's non-volatile storage. The development team should leverage standard local storage packages, such as `shared_preferences`, to write and retrieve this configuration string asynchronously. Upon application initialization, the system must independently load the saved URL string into the active memory state to prepare the networking client for immediate operation. Should a user attempt to transmit data prior to configuring a valid URL, the application must intercept the action and display a localized alert prompting immediate configuration.
